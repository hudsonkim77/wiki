# 배포 관리

## 개요
이 문서는 여러 변경 건을 묶어 하나의 배포 단위로 계획·빌드·테스트·배포하는 기준을 안내합니다.

## 목적
- 개별 변경 티켓들을 배포 단위로 묶어 배포 일정과 범위를 통제합니다.
- 배포 실패 시 어떤 변경 묶음을 롤백해야 하는지 추적 가능하게 합니다.

## 배포 처리 프로세스
- (작성 예정) 배포 계획 → 대상 변경 확정 → 빌드 → 테스트 → 배포 → 배포 후 검증 단계별 절차

## 필드 정의 (DEPLOY 테이블)
[1_변경관리](../1_변경관리/README.md)의 CHANGE 테이블과 연결되는 별도 테이블로 설계합니다.

| 필드명 | 설명 | 비고 |
|---|---|---|
| DEPLOY_ID | 배포 고유 ID | **PK**. 예: `DEP_20260711_001` |
| DEPLOY_NAME | 배포명 | |
| DEPLOY_TYPE | 배포 유형 | 정기/긴급/롤백 |
| DEPLOY_STATUS | 진행 상태 | 계획/빌드/테스트/배포완료/롤백 |
| PLANNED_DT | 배포 계획 일시 | |
| DEPLOYED_DT | 실제 배포 일시 | |
| OWNER_TEAM | 담당 부서 | |
| DESC | 배포 범위/내용 설명 | |

## 변경관리와의 연결
- 배포 1건은 여러 변경 티켓을 묶을 수 있고, 변경 티켓 1건도 재배포·롤백 과정에서 여러 배포에 포함될 수 있어 **N:M** 관계입니다.
- 실제 연결은 **`DEPLOY_CHG_MAP` 중간 테이블**을 통해 관리하며, 데이터 파일은 [배포이력/](배포이력/README.md) 폴더에 둡니다.
