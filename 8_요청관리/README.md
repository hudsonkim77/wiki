# 요청 관리

## 개요
이 문서는 장애가 아닌 일반적인 사용자 요청(계정 발급, 권한 변경, 장비 요청 등)의 접수·처리 기준을 안내합니다. `RAW/erd-참고용.png`의 "통합 ITSM 13단계 운영 흐름" 상 **REQ(요청관리)** 항목에 대응합니다.

## 목적
- 장애(비정상 상태 복구)와 성격이 다른 표준화된 요청을 별도로 접수·추적합니다.
- 처리 시간을 측정해 서비스수준관리(SLA)의 기초 데이터로 활용합니다.
- 처리 중 문제가 확인되면 장애로 확대(에스컬레이션) 가능하도록 장애관리와 연결합니다.

## 요청 처리 프로세스
- (작성 예정) 요청 접수 → 승인(필요 시) → 처리 → (필요 시 장애 확대) → 완료 통보 단계별 절차

## 필드 정의 (REQUEST 테이블)

| 필드명 | 설명 | 비고 |
|---|---|---|
| REQ_ID | 요청 고유 ID | **PK**. 예: `REQ_20260711_001` |
| REQ_TITLE | 요청 제목 | |
| REQ_TYPE | 요청 유형 | 계정발급/권한변경/장비요청/정보조회/기타 |
| REQUESTER_ID | 요청자 ID | FK → 구성관리 CI.CI_ID (HUMAN 유형) |
| STATUS | 진행 상태 | 접수/처리중/완료/반려 |
| REQUESTED_DT | 요청 접수 일시 | |
| COMPLETED_DT | 처리 완료 일시 | |
| ASSIGNED_TEAM | 처리 담당 부서 | |
| DESC | 요청 상세 내용 | |

## 구성관리와의 연결
- 요청 1건은 여러 CI에 영향을 줄 수 있고(예: PC+계정 동시 발급), CI 1건도 여러 요청과 연결될 수 있어 **N:M** 관계입니다.
- 실제 연결은 **`REQUEST_CI_MAP` 중간 테이블**을 통해 관리하며, 데이터 파일은 [요청이력/](요청이력/README.md) 폴더에 둡니다.

## 장애관리와의 연결
- 참고 그림의 REQ→INC 흐름("요청ID → Incident ID")대로, 처리 중인 요청이 장애로 확대될 수 있어 **N:M** 관계로 봅니다(한 요청이 장애로 이어지거나, 여러 요청이 동일 장애의 원인으로 모일 수 있음).
- 실제 연결은 **`REQUEST_INCIDENT_MAP` 중간 테이블**로 관리합니다([요청이력/](요청이력/README.md)).
