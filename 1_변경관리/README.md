# 변경 관리

## 개요
이 문서는 시스템/서비스에 대한 변경 요청, 승인, 적용 이력을 관리하는 기준을 안내합니다.

## 목적
- 변경으로 인한 장애를 예방하고, 변경 사항을 추적 가능하게 관리합니다.

## 변경 요청 프로세스
- (작성 예정) 요청 → 검토 → 승인 → 적용 → 사후 검증 단계별 절차

## 필드 정의 (CHANGE 테이블)
`3_구성관리`의 CI 마스터(`CFG.csv`)에 존재하는 `CHG_TICKET_ID` 필드를 근거로, 변경 요청 자체를 관리하는 별도 테이블을 아래와 같이 설계합니다.

| 필드명 | 설명 | 비고 |
|---|---|---|
| CHG_TICKET_ID | 변경 티켓 ID | **PK**. 예: `CHG-2026-001`, `CHG_20260302_001` (원본 데이터에 표기 형식이 2가지 혼재 → 정합화 필요) |
| CHG_TITLE | 변경 제목 | |
| CHG_TYPE | 변경 유형 | 예: 정기/긴급/표준 |
| REQUEST_TEAM | 요청 부서 | |
| REQUESTER_ID | 요청자 ID | FK → 구성관리 CI.CI_ID (HUMAN 유형) |
| APPROVER_ID | 승인자 ID | FK → 구성관리 CI.CI_ID (HUMAN 유형) |
| CHG_STATUS | 진행 상태 | 예: 요청/승인/적용/완료/롤백 |
| PLANNED_DT | 적용 계획 일시 | |
| APPLIED_DT | 실제 적용 일시 | |
| ROLLBACK_YN | 롤백 여부 | |
| IMPACT_LEVEL | 영향도 | |
| RELATED_DESC | 변경 상세 내용/사유 | |

## 구성관리와의 연결
- 변경 1건은 여러 CI에 영향을 줄 수 있고, CI 1건도 여러 변경 이력을 가질 수 있어 **N:M** 관계입니다.
- 실제 연결은 CHANGE 테이블과 CI 테이블을 직접 잇지 않고, **`CHANGE_CI_MAP` 중간 테이블**을 통해 관리합니다. 설계 근거(ERD)는 [3_구성관리](../3_구성관리/README.md)에 기록되어 있고, 실제 데이터 파일은 변경 도메인 하위의 [변경히스토리/](변경히스토리/README.md) 폴더에 둡니다.

## 생성된 테이블 파일
| 파일 위치 | 설명 | 건수 |
|---|---|---|
| `CHANGE.csv` (이 폴더) | 위 필드 정의를 구현한 변경 티켓 마스터. `../RAW/CFG.csv`의 `CHG_TICKET_ID`에서 고유값만 추출한 9건 + [5_ITSM관리](../5_ITSM관리/README.md) 통합관리대시보드 관련 건(`CHG_20260711_001`~`003`) 3건 | 12건 (기존 9건은 제목/유형 등 세부 항목이 원본 데이터에 없어 공란, 추후 보완 필요. `CHG_20260711_001`~`003`은 전 필드 기입) |
| [`변경히스토리/CHANGE_CI_MAP.csv`](변경히스토리/README.md) | CHANGE-CI 중간(연결) 테이블. 어떤 CI가 어떤 변경 티켓에 연결되어 있는지 실제 매핑 | 64건 |
